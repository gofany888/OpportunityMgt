# 2026-03-31 设计变更说明（工作台首页与商机详情页）

## 1. 本次变更范围

本次设计变更主要覆盖以下两块：

1. 工作台首页
   - 核对差异表格
2. 商机详情页
   - 头部提交区
   - 基本信息卡片中的下拉字段
   - 商机环节执行里程碑管理
   - 草稿态商机进度执行轨迹

本说明用于开发实现与验收，不要求开发自行“对比页面猜需求”。请以本说明中的“硬规则”和“验收点”为准。

---

## 2. 设计目标

本轮修改的核心目标是：

1. 统一商机详情页的编辑语义
   - 默认静态展示
   - hover 出编辑图标
   - 点击进入编辑
   - 点击外部恢复静态展示
   - 改过后显示小红点
2. 明确里程碑管理中的计划时间、实际时间、附件状态和提交保护逻辑
3. 降低表格的视觉噪音，提高信息密度下的可读性
4. 避免上传中、联动中、空值同步中的业务冲突

## 3. 工作台首页：核对差异表格

### 3.1 改了什么

1. 数据区加入“隐形垂直斑马列”引导
   - 仅BG台账包含
   - 收入
   - 毛利
   这几组数据列在正文和合计区加入很轻的灰底引导。
2. 表头统一增加轻灰底
   - 表头背景色使用 `#FAFBFC`
3. 多表头表头区不加斑马列底色
   - 只在数据区和合计区做列引导

### 3.2 为什么这样改

1. 降低大宽表的扫读压力
2. 不靠重边框切割，而是用更轻的列引导帮助定位
3. 多表头场景下，表头再叠列底色会让层级变乱，因此只保留中性轻灰底

### 3.3 硬规则

1. 表头背景统一为轻灰底，不允许纯白表头
2. 垂直斑马列只出现在数据区，不出现在多表头层级区
3. 列引导只能是低对比灰度，不允许重色块

### 3.4 验收点

1. 表头视觉上是一个完整的轻灰层
2. `仅BG台账包含 / 收入 / 毛利` 这三组列在正文里能看到轻微列引导
3. 多表头标题区不出现斑马底色干扰

## 4. 商机详情页：头部与统一提交保护

### 4.1 改了什么

1. 页面标题支持可见即所得编辑
   - hover 出编辑图标
   - 点击进入输入框
   - 点击外部保存回静态文本
   - 改过后显示小红点
2. 顶部提交按钮增加“附件上传中禁用”保护
   - 只要里程碑中有任意附件处于上传中，顶部提交按钮禁用
   - tooltip 提示：`附件上传中，请等待完成后提交`
   - tooltip 从按钮下方出现

### 4.2 为什么这样改

1. 标题与页面其他字段编辑体验保持一致
2. 避免附件尚未完成上传时，页面进入“半提交”状态

### 4.3 硬规则

1. 上传中时，顶部提交按钮必须不可点
2. 上传完成后，按钮自动恢复可提交
3. tooltip 需出现在按钮下方，不允许被页面顶部裁切

### 4.4 验收点

1. 任何一个里程碑子项上传附件时，顶部提交按钮立即禁用
2. 上传完成后自动恢复
3. 鼠标移到禁用按钮上，能看到下方提示

## 5. 商机详情页：基本信息字段编辑方式

### 5.1 改了什么

以下字段已改成下拉框编辑，而不是文本输入：

1. 销售单元
   - 北部分公司
   - 北京分公司
   - 集成能力中心
   - 南部分公司
   - 事业部
   - 西部分公司
2. 合同状态
   - 已撤销
   - 已定稿
   - 已盖章
   - 已归档
   - 已签约
3. 关联交易类型
   - IT服务
   - 物资采购
   - 与关联方的非关联交易事项
4. 财务分类
   - 国内工程技术服务收入
   - 国内技术委托开发收入
   - 国内软件技术服务收入
   - 国内系统集成收入
   - 国内自主开发软件收入
   - 国外软件技术服务收入
   - 国外自主开发软件收入
5. 项目属性
   - 资源池
   - 非资源池
6. 招标类型 / 投标类型（与里程碑采购环节联动）

### 5.2 清空规则

1. 除 `二级部门` 外，其它这些下拉框都允许清空
2. 清空后的静态展示统一为 `--`
3. 招标类型清空后，不仅当前字段显示 `--`，还需要同步影响里程碑采购环节子环节

### 5.3 为什么这样改

1. 这些字段都是枚举值，不适合自由文本输入
2. 清空能力是业务真实需要，不能强行保留旧值
3. 招标类型与采购环节是强业务联动，不能只改一处不改另一处

### 5.4 硬规则

1. `二级部门` 不允许清空
2. 其他枚举型字段允许清空，并显示 `--`
3. 招标类型清空必须视为“有效空值”，不能被旧值回填

### 5.5 验收点

1. 销售单元、合同状态、财务分类等点击后均出现下拉框
2. 非二级部门字段均可清空
3. 清空招标类型后：
   - 信息卡显示 `--`
   - 里程碑右上角同步为 `--`
   - 采购环节子环节切换到空值默认规则

## 6. 商机详情页：商机环节执行里程碑管理

### 6.1 招标类型与采购环节联动

#### 改了什么

在里程碑卡片右上角新增一个与基本信息区联动的“招标类型”字段，默认静态、点击可编辑。

选择不同招标类型时，`L4. 采购环节` 子环节立即切换：

1. 公开招标
   - 采购需求
   - 采购方案
   - 公告发布
   - 文件投递截止及递交
   - 候选人公示
   - 采购结果
2. 公开询比
   - 采购需求
   - 采购方案
   - 公告发布
   - 文件投递截止及递交
   - 候选人公示
   - 采购结果
3. 直接采购（单一来源）
   - 采购需求
   - 采购方案
   - 公告发布
   - 候选人公示
   - 采购结果
4. 框架下订单
   - 采购需求
   - 采购方案
   - 订单生成
5. 为空
   - 采购需求
   - 采购方案
   - 采购结果

#### 硬规则

1. 基本信息区和里程碑右上角的招标类型必须双向同步
2. 清空后采购环节子项必须切换为空值默认规则
3. 采购环节标题有值时显示 `L4. 采购环节（xxx）`，无值时显示 `L4. 采购环节`

---

### 6.2 父环节展开规则

#### 改了什么

1. 默认只有“进行中”的父环节展开
2. 其他父环节默认收起

#### 验收点

1. 页面初始进入时，只展开当前进行中的阶段
2. 已完成、计划中父环节默认收起

---

### 6.3 计划周期规则

#### 改了什么

1. 计划周期继续使用 Ant Design Vue 的 `RangePicker`
2. 中间分隔符使用组件官方 `separator` 机制，不直接改控件内部默认图标逻辑
3. 分隔符现在是自定义的 `~`，并做视觉补偿，使其更靠近开始和结束的中间空隙

#### 父环节计划时间规则

1. 父环节 `已完成`
   - 不可编辑
2. 父环节 `进行中` 且有子环节
   - 不可编辑
   - 显示值按子环节汇总
3. 父环节 `计划中` 或无子环节
   - 可编辑

#### 父环节计划时间汇总逻辑

1. 开始时间
   - 优先取第一个子环节的开始时间
   - 如果没有，则回退到父环节原来自己填的开始时间
2. 结束时间
   - 优先取最后一个子环节的结束时间
   - 如果没有，则回退到父环节原来自己填的结束时间

即：
- `start = firstChild.start || parent.start`
- `end = lastChild.end || parent.end`

#### 子环节计划时间规则

1. 子环节计划时间依旧可编辑
2. 计划开始时间不能小于上一环节的计划结束时间
3. 若选择非法范围，本次选择不生效，并显示红色提示：
   - `计划开始时间不能小于上一环节结束时间`

#### 为什么这样改

1. 父环节与子环节不能同时作为两个冲突的数据源
2. 父环节前期录入的计划值不能因为子环节尚未细化而丢失
3. 采用“子环节优先、父环节兜底”最符合实际排期过程

#### 硬规则

1. 进行中的父环节若有子环节，计划时间必须只读
2. 计划中的父环节仍可编辑计划时间
3. 父环节汇总逻辑必须为“子环节优先、父环节兜底”
4. 计划开始时间不得早于上一环节结束时间

#### 验收点

1. 进行中父环节不出现计划时间控件
2. 计划中父环节仍可编辑
3. 子环节填写后，父环节计划时间显示会跟着变化
4. 子环节未填全时，父环节能继续显示自己原来的计划边界

---

### 6.4 实际时刻规则

#### 改了什么

1. 已完成状态的环节，计划时刻和实际时刻都不可编辑
2. 父环节若有子环节，实际时刻按子环节汇总
   - 开始 = 第一个子环节开始
   - 结束 = 最后一个子环节结束
3. 若子环节填写了实际结束时间，下一个环节默认开始时间自动为 `上一个结束 + 1 天`
4. 若当前已经是最后一个子环节，则继续看下一个父环节：
   - 下一个父环节有子环节：自动展开，并带起第一个子环节的实际开始时间
   - 下一个父环节无子环节：直接在该父环节上进入实际时刻填写逻辑
5. 若清空某一环节的实际结束时间，只回收系统自动带入的下一环节默认值，不误删用户手填值

#### 校验规则

1. 实际开始时间不能小于上一环节结束时间
2. 结束时间必须大于或等于开始时间
3. 非法时显示行内红色提示

#### 硬规则

1. 已完成环节不可编辑计划时间和实际时间
2. 有子环节的父环节实际时刻必须只读并汇总
3. 自动带入的值可以回收，用户手填值不自动清空

---

### 6.5 附件列交互

#### 改了什么

附件列不再使用长文字按钮，而改成轻量图标模式：

1. 未上传
   - 显示一个上传图标
2. 已上传
   - 显示两个图标：
     - 修改附件
     - 下载附件
3. 不适用 / 计划中
   - 显示 `--`

#### 上传交互

1. 点击上传附件
   - 直接弹出操作系统文件选择框
2. 用户选择文件后
   - 当前行进入 `uploading`
   - 显示一个小圆形进度环
   - 下载按钮显示但禁用
3. 约 2 秒后
   - 进度完成
   - 当前行切换为已上传态
   - 显示：修改附件 + 下载
4. 点击修改附件
   - 也是同样流程：重新选择文件 -> 上传中 -> 切回已上传态

#### 父环节附件状态

父环节不显示上传、修改、下载按钮，只显示三种汇总图标：

1. 全部未上传
2. 部分已上传
3. 全部已上传

说明通过 tooltip 展示，不直接显示文字。

#### 为什么这样改

1. 文字按钮在表格里太臃肿
2. 图标更适合高密度清单
3. 上传过程需要有中间态，不能“点一下就瞬间成功”
4. 父环节承担的是汇总语义，不承担附件操作语义

#### 硬规则

1. 上传 / 修改附件必须走系统文件选择框
2. 上传中必须显示小圆形进度环
3. 上传中下载按钮必须不可点击
4. 父环节不得出现上传/修改/下载按钮
5. 计划中的环节显示 `--`，不出现上传图标

#### 验收点

1. 点击上传图标后能打开系统选文件框
2. 选择文件后进入上传中态
3. 上传中显示进度环，且下载按钮禁用
4. 2 秒后切换成修改附件 + 下载图标
5. 父环节只显示汇总图标，不显示操作按钮

## 7. 商机详情页：草稿态商机进度执行轨迹

### 7.1 改了什么

1. 上方输入区按钮文案从“提交进展”改成 `加入草稿`
2. 点击后，不是直接正式提交，而是追加到底部草稿列表
3. 新草稿插入在第一条（最新在前）
4. 点击删除会真正删除草稿
5. 草稿态展示统一为：
   - `未提交 · 当前用户`
6. 展示态保留文本换行
7. 只有内容真实超过 3 行时，才显示 `点击阅读全文`

### 7.2 为什么这样改

1. 这块当前是草稿流，不是正式提交流，“提交进展”语义不准确
2. 草稿应该强调“未提交”状态，而不是伪时间
3. 文本里的回车必须保留，否则展示内容和编辑内容不一致

### 7.3 硬规则

1. 草稿态 meta 统一显示 `未提交 · 当前用户`
2. 新草稿插入第一条，不追加到末尾
3. 删除操作必须真实生效
4. 回车必须在展示态保留
5. 只有超过 3 行才显示 `点击阅读全文`

### 7.4 验收点

1. 输入内容后点击 `加入草稿`，底部第一条新增草稿
2. 草稿显示 `未提交 · 当前用户`
3. 删除按钮可删除草稿
4. 有回车的文本在展示中仍有回车
5. 三行以内不出现“点击阅读全文”

## 8. Ant Design Vue 组件处理原则（本次落实）

### 8.1 本次案例

里程碑计划周期中间分隔符调整时，采用了 Ant Design Vue `RangePicker` 官方支持的 `separator` 机制，而不是继续长期覆盖默认内部图标结构。

### 8.2 统一规则

后续开发实现时，若涉及 Ant Design Vue 组件调整，必须遵循：

1. 先查官方文档或本地源码
2. 优先使用组件提供的 `prop / slot / token / size / variant / separator`
3. 只有官方机制无法满足时，才允许最小化 `:deep()` 覆盖
4. 禁止先通过试错式 CSS 去硬改控件内部 DOM

### 8.3 为什么要这样做

1. 组件升级兼容性更好
2. 不容易出现“样式看起来生效，其实没有真正命中”的情况
3. 减少后期维护成本

## 9. 给开发的实现建议

### 9.1 优先级建议

建议开发按以下顺序实现：

1. 里程碑计划时间/实际时间/招标类型联动规则
2. 附件上传中状态与顶部提交流程保护
3. 基本信息枚举字段下拉化与清空规则
4. 草稿态进度执行轨迹交互
5. 工作台首页核对差异表格视觉优化

### 9.2 不建议开发自己猜的内容

以下内容不要通过“看页面差异自己猜”来实现，应严格按本说明：

1. 进行中父环节计划时间是否可编辑
2. 招标类型为空时采购子环节规则
3. 上传中是否允许提交
4. 父环节附件是否显示按钮
5. 草稿态到底是时间语义还是未提交语义

---

## 10. 建议验收方式

建议验收时按以下清单逐条检查：

1. 招标类型切换与采购环节子项联动
2. 招标类型清空后的默认规则
3. 父环节计划时间的只读/汇总逻辑
4. 实际时刻的顺推、回收、校验
5. 附件上传 -> 进度环 -> 修改/下载 的完整链路
6. 上传中顶部提交按钮禁用
7. 草稿加入、删除、换行与阅读全文逻辑
8. 工作台首页核对差异表格表头和列引导样式
