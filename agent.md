# B端企业级页面最佳设计规范 (Modern Soft Enterprise)

这份文档总结了对于复杂业务数据页面（如财务报表、经营看板等）重构中得出的核心设计规范与最佳实践。旨在平衡高密度数据阅读的严谨性与现代 B 端系统的亲和力。

## 1. 全局设计基调：冷静、专业、透气
*   **企业级底色与卡片**：放弃过度的渐变色与抢眼修饰。使用标准的后台冷灰 (`#f0f2f5`) 作为页面大底色，搭配纯白 (`#ffffff`) 的悬浮卡片。通过克制的弥散阴影 (`box-shadow`) 建立页面的立体层级感。
*   **现代软和风 (Modern Soft)**：摒弃传统的锐利直角。全局核心的承载容器（如大卡片、表格外部包裹块）统一使用 `16px` 大圆角（例如 Tailwind 的 `rounded-2xl`），而内部小组件、标签采用适中的 `8px` (`rounded-lg`)。大圆角能极大削弱复杂数据的心理压迫感，呈现类似 Notion、飞书等现代 SaaS 产品的无压交互体验。

## 2. 表格视觉结构：去线化与横向引导
*   **彻底打破“牢笼感” (去外框线)**：在现代企业级大表格中，**坚决去除表格最外侧的左右实线边框**（包括表头、数据行、总计行）。让表格两侧“透气”，避免数据被框死。
*   **垂直业务域划分**：复杂的对账表格往往需要横跨多个口径（如经营侧、推盘侧、差异分析侧）。对于表头 (`thead`)，可使用柔和的浅色块背景（如极浅绿、浅蓝、浅红）进行逻辑分区，结合对应的色彩基调约束当前业务块。
*   **水平视线引导**：表体 (`tbody`) 数据采用轻量的横向底部分界线（`border-b`)，辅以整行 hover 状态的淡色遮罩（如 20% 透明度的同色系背景），引导视线从左向右平滑扫视。切忌在中间加入大量全框实线。

## 3. 表脚 (Footer) 总计行的稳固感
*   **双线兜底 (Border-Y)**：作为表格数据的“终点结账行”，不需要左右侧封口，但必须使用带有物理切割感的上下加粗边框 (如 `border-y-[3px]`) 对表格进行收尾承托，防止视觉下流。
*   **高对比度提权**：总计行是结论所在。需加深该行的整体底色，并通过字号放大 (`text-lg`) 和字重加粗 (`font-bold`) 从上方密密麻麻的常规数据中“跳”出来，做到一眼抓取结论。

## 4. 极致的数据对齐：等宽数字字体
*   高精度的对账表，数字必须上下能够对齐阅读。全局运用**等宽数字排版特（Tabular Nums）**。
*   **实现方案**：通过 CSS 定义 `font-variant-numeric: tabular-nums;` 让任何西文字体（例如 Inter 或 Helvetica）呈现小数点和个位、十位严格对齐的排版，这对于千万、百万级别的资金校对至关重要。

## 5. Ant Design Vue 组件优先原则

*   **能用 antdv 就用 antdv**：所有 UI 元素优先使用 Ant Design Vue 4.x 组件，禁止手写原生 HTML 再加 Tailwind 类名来平替已有的 antdv 组件。
*   **常用组件映射表**：

    | 场景 | 使用 antdv 组件 | 禁止做法 |
    |------|----------------|---------|
    | 页面布局 | `a-layout` / `a-layout-sider` / `a-layout-header` / `a-layout-content` | 手写 `<aside>` / `<header>` + Tailwind |
    | 侧边栏导航 | `a-menu` / `a-menu-item`（支持 `selected-keys` 自动高亮） | 手写 `<a>` / `<router-link>` 列表 |
    | 按钮 | `a-button`（支持 `type`、`:icon`、`loading` 等） | `<button class="...">` |
    | 标签/徽章 | `a-tag`（`color` 属性）/ `a-badge` | 手写 `<span>` + 背景色 |
    | 数据统计 | `a-statistic`（内置 `prefix`、`suffix`、`value-style`） | 手写 `<div>` 拼数字 |
    | 数据表格 | `a-table`（`columns.children` 分组表头、`#summary` 总计行） | 原生 `<table>` |
    | 搜索框 | `a-input`（`prefix` / `suffix` 插槽） | `<input>` + 图标 |
    | 提示/警告 | `a-alert`（`type="error/warning/info"`） | 手写红色 div |
    | 间距 | `a-space`（`size` 属性） | 手写 `gap` |
    | 分割线 | `a-divider` | `<hr>` 或 `border-b` |
    | 文字排版 | `a-typography-title` / `a-typography-text` / `a-typography-paragraph` | 原生 `<h1>` ~ `<p>` |
    | 栅格布局 | `a-row` / `a-col`（`:gutter`、`:xs`/`:sm`/`:lg`/`:xl` 响应式） | Tailwind `grid` |
    | 卡片容器 | `a-card`（`:bordered="false"`、`#title` 插槽） | 手写 `<div class="bg-white rounded...">` |
    | 头像 | `a-avatar` | `<img class="rounded-full">` |
    | 图标 | `@ant-design/icons-vue`（按需 import） | Font Awesome / 手写 SVG |

*   **a-table 使用规范**：
    *   分组表头用 `columns` 中的 `children` 嵌套，而非手写 `rowspan` / `colspan`。
    *   单元格差异化样式用 `customHeaderCell` / `customCell` 回调返回内联 `style`。
    *   单元格内容定制用 `#bodyCell="{ column, record }"` 插槽按 `column.dataIndex` 分发。
    *   总计行用 `#summary` 插槽 + `a-table-summary-row` + `a-table-summary-cell`。
    *   行点击用 `customRow` 返回 `{ onClick }` 而非在 `<tr>` 上绑定事件。
    *   不需要横向滚动时**禁止**设置 `scroll: { x }` 和 `fixed`。

## 6. 中文排版反模式（必须避免）

*   **禁止 `italic`（斜体）**：中文字形未设计斜体变体，CSS `font-style: italic` 只会让字形产生不自然的错切变形。
*   **禁止 `letter-spacing`（字间距）**：中文字符本身已含内建间距，额外的 `tracking-widest` / `tracking-wide` 会破坏阅读节奏。
*   **禁止 `uppercase` / `lowercase`**：仅对拉丁字母有效，对中文无意义且易造成混淆。
*   **字号规范**：页面标题 `16px`、正文 `14px`（antdv 默认）、辅助说明 `13px`、分组标签/脚注 `12px`。

## 7. 数据分离与文件结构

*   **所有静态/Mock 数据必须外置**：将页面中的硬编码数据、菜单配置、表格列定义等提取到 `src/data/` 目录下的独立 JS 文件中。
*   **命名约定**：`{页面名}Data.js`，如 `menuData.js`、`dashboardData.js`、`reportCenterData.js`、`businessOpportunityData.js`。
*   **导出格式**：使用具名 `export const`，禁止 `export default`，便于 tree-shaking 和按需引用。
*   **未来扩展**：数据文件格式与后端 API 返回结构保持一致，方便后续从 mock 切换为真实接口调用。

## 8. 布局架构：Teleport 动态注入

*   **全局 Header 的动态内容**：`BaseLayout.vue` 的 Header 中预置两个 Teleport 锚点：
    *   `#header-title-portal` — 标题区（返回按钮 + 页面标题 + 状态标签）
    *   `#header-actions-portal` — 操作区（按钮组）
*   **子页面注入方式**：各子页面通过 `<Teleport to="#header-title-portal">` 将自己的标题/操作注入全局 Header，需使用 `v-if="isMounted"` 确保目标DOM已就绪。
*   **路由 meta 配置**：在 `router/index.js` 中为每个路由设置 `meta.title`、`meta.customTitle`、`meta.customActions`，`BaseLayout.vue` 根据 meta 决定是否显示默认标题/操作。

## 9. 间距与设计 Token

*   **间距倍数**：所有间距（margin、padding、gap）严格使用 **8px 倍数**：`8`、`12`（1.5倍，仅限紧凑场景）、`16`、`24`、`32`、`40`。
*   **色彩 Token**：
    *   主色：`#1677ff`（antdv 标准蓝）
    *   页面底色：`#f0f2f5`
    *   卡片：`#ffffff`
    *   成功/经营类：`#52c41a` / `#f6ffed` / `#389e0d`
    *   警告/关注类：`#fa8c16` / `#fff7e6`
    *   危险/差异类：`#ff4d4f` / `#fff2f0` / `#cf1322`
    *   信息/预测类：`#1677ff` / `#e6f4ff` / `#0958d9`
*   **卡片圆角**：外层大容器 `16px`（`border-radius: 16px`），内部小组件 `8px`。
*   **阴影层级**：统一使用克制的三层弥散阴影：
    ```css
    box-shadow: 0 1px 2px rgba(0,0,0,0.03),
                0 1px 6px -1px rgba(0,0,0,0.02),
                0 2px 4px rgba(0,0,0,0.02);
    ```

## 10. CSS 样式分离 (关注点分离)

*   **隔离通用与个性样式**：为了避免 Vue 单文件组件 (SFC) 代码过于臃肿，以及遵循关注点分离的原则，所有的业务组件（尤其是重型报表表格、卡片面板等）的局部样式必须抽离。
*   **提取规则**：将原本写在 `.vue` 文件内部的 `<style scoped>...</style>` 块里的 CSS 样式，完整提取到该组件同级目录下的 `styles/` 文件夹中。
*   **命名规范**：提取的 css 文件名应与其对应的 Vue 组件名保持绝对一致。例如 `DiffTable.vue` 对应的样式文件必须为 `styles/DiffTable.css`。
*   **引入方式**：在 Vue 组件中，通过 `src` 属性以外链的方式引入这个局部作用域样式文件：
    ```vue
    <style scoped src="./styles/ComponentName.css"></style>
    ```
*   **Tailwind 结合**：对于轻量级的布局、间距、颜色调整，依然鼓励在 `<template>` 中直接使用纯 Tailwind CSS 工具类。只有当遇到无法用 Tailwind 简洁表达的复杂复合选择器（如对 antdv 第三方组件库深层 DOM 节点 `:deep()` 的覆盖重写、复杂的伪类交互态、特化的动画定义等），才必须提取进入这套独立的 `.css` 文件规范中。直接使用纯 Tailwind 的组件不需要创建空的 `.css` 文件和 `<style>` 标签。

## 11. 通用实现补充规范


*   **复杂表格展开规范**：当业务表格需要自定义展开按钮的位置和样式时，优先隐藏 antd 默认展开图标，用业务化按钮替代；若树表默认机制不适配设计，可使用“拍平数据 + 条件渲染子行”的方式实现展开收起。
*   **日期控件中文化**：中文业务页面中的日期选择器，必须同时设置 antd 组件 locale 和底层日期库 locale；占位文案统一使用中文，如“开始日期 / 结束日期”。
*   **无横向滚动优先**：不需要横向滚动的大表格，禁止使用 `scroll.x` 兜底，应优先通过固定布局、压缩列宽、缩减 padding、精简辅助信息来收敛宽度。
*   **信息层级字重**：普通字段值默认 `400`，分区标题 `500`，关键状态/金额/结论 `600`；禁止将大面积普通信息统一加粗，避免页面层级失真。
*   **分页控件样式**：页码与前后翻页按钮分开处理；前后按钮优先使用轻量描边样式，不做笨重灰块；总数与范围说明使用辅助信息层级。

## 12. 经营看板图表与表格规范（以本节为准）

> 若本节与前文存在冲突，涉及经营看板、序时图、部门经营表、核对差异表等场景时，统一以本节为准。

### 12.1 卡片跳转与页面返回
*   **摘要卡片不整卡跳转**：首页顶部摘要卡只负责信息展示，不应整张卡片都可点击跳转，避免误触与阅读干扰。
*   **只保留明确跳转入口**：跳转行为只挂在卡片右上角的跳转按钮上；按钮默认可隐藏，在 hover 卡片时再显现。
*   **跳转按钮反馈要轻但明确**：按钮 hover 时可使用品牌色加深、淡色圆角底、轻微上浮等方式提示可点击，但不能喧宾夺主。
*   **返回逻辑使用显式路由来源**：返回按钮是否显示，应依据 Vue Router 的显式来源参数（如 `query` 或 `state`）判断，禁止依赖浏览器历史长度或模糊的 `window.history` 猜测。

### 12.2 图表表达规范
*   **当前主视图聚焦，tooltip 全量补充**：柱状图主画面可通过 tab 在“收入 / 采购 / 毛利”之间切换，但鼠标 hover 到某个月份时，tooltip 应同时展示该月的全部关键口径，避免用户来回切 tab 对比。
*   **实绩与预测要明确分区**：实绩与预测需同时通过月份位置、分隔线、标签说明体现边界，`预测` 标签应放在分隔线附近且避开柱体本身，不能压在柱子上。
*   **预测纹理必须柔和**：斜向纹理不得使用纯白高对比线条，应使用同色系浅色或半透明白，让纹理融入底色而不是跳出底色。
*   **图例优先横向并排**：若图例项较少，应优先横向排列，减少头部高度占用。
*   **tab 必须具备控制感**：图表上方的 tab 不能过小，字号、字重、内边距要足以体现“可操作主控件”身份。

### 12.3 表格信息架构
*   **单位集中说明，不在每列表头重复**：金额和个数单位应统一放在表格右上角的小字说明中，例如 `单位：金额(万元) / 个数`；表头本身只保留 `金额`、`个数` 等核心字样。
*   **最底层表头必须强制触底对齐**：多级表头中，所有真正承载数据的最底层子表头（如 `个数 / 金额`）必须落在同一条水平基线上；必要时可增加只参与布局、不参与视觉的桥接层。
*   **大数据块可用轻量竖线分隔**：当表头横跨多个业务块时，可在大块之间增加 `1px solid #E5E6EB` 级别的细微垂直分割线，但只用于分组边界，不切碎整张表。
*   **数字列对齐需分场景处理**：纯财务清单型表格可直接整列右对齐；多级表头或“表头中线需与数字中线关联”的报表，应用“单元格居中承载数字槽，槽内数字右对齐”的模型，不能一句“全部右对齐”粗暴处理。
*   **金额统一两位小数**：金额数据默认保留两位小数；若某列业务上明确不展示，则使用 `--`，不要混用空字符串或 `0` 代替。

### 12.4 数字视觉优先级
*   **普通数字默认不加粗**：除差异行、汇总行外，普通数据数字统一使用常规字重，不可大面积加粗。
*   **颜色只服务真实状态**：表头、汇总行、说明文字统一使用高级中性色（深灰/黑灰）；红绿等强调色只应用在真正表达状态的单元格值上，例如正负差异、利润正负等。
*   **差异红绿语义必须跟业务口径走，不可机械套“正绿负红”**：若业务定义为 `BG > EIS` 代表风险或异常，则正差异必须用红色、负差异用绿色；同一页面内的表格、tooltip、汇总都必须保持一致。
*   **零值必须降噪**：`0` 或 `0.00` 统一使用浅灰色弱化，推荐 `#C9CDD4`，避免和有效数值抢视觉。
*   **差异行可以强调，但要克制**：差异值可使用红绿颜色，并允许适度字重提升；差异整行可配合一层很浅的底色帮助识别，但不能过重。
*   **汇总行强调方式以收口为主**：汇总行优先使用上边界线 + 浅底色 + 较高字重，不使用上下双粗线“夹住”整行。

### 12.5 数字下钻交互
*   **可点击数字默认不加下划线**：数字可下钻时，不要通过常驻下划线提示，避免界面发花。
*   **hover 使用品牌色提示**：用户移入可点击数字时，再切换为品牌色，并配合 `pointer` 光标与轻量 tooltip（如 `查看明细`）提示可下钻。
*   **个数与金额遵循同一套点击规则**：只要业务允许下钻，个数和金额都应使用同样的交互反馈，不应一套可点、一套不可点。
*   **合计行若支持下钻，必须继承与正文一致的点击语义**：汇总/合计行不是静态收口时，也必须复用同样的 hover、tooltip、颜色与格式规则，不能正文可点、合计失效。

### 12.6 看板型分组表推荐写法
*   **分组标题可承接差异数据**：在经营看板类对比表中，可将差异数据上收至分组标题行（如 `收入差异 / 采购差异 / 毛利差异`），替代单独再放一行"差异"。
*   **分组标题支持展开收起**：若分组下还有 `EIS系统 / BG台账` 等来源行，可通过分组前的小型 `+ / -` 控件展开收起，提高高密度场景下的可读性。
*   **实绩列与预测列要拉开层次**：实绩列可使用更明显的灰色背景统一成组，帮助用户快速区分"已经发生"和"后续预测"。

## 13. 多层级固定表头/搜索栏的最佳实践 (Multi-Layer Sticky Headers)

> 适用于台账中心、数据列表等需要滚动时固定顶部导航、搜索栏、表头的场景。

### 13.1 架构分层策略

当页面由 **Tabs 导航 → 搜索/筛选栏 → 表头 → 数据行** 组成时，采用三层 sticky 叠加方案：

| 层级 | 元素 | sticky top | z-index | 背景 |
|------|------|-----------|---------|------|
| Layer 1 | 顶部 Tabs | `0` | `20` | `#ffffff`（纯白不透明） |
| Layer 2 | 搜索/操作栏 | `var(--tabs-height)` | `15` | 页面底色 `var(--page-background)` |
| Layer 3 | 表格表头 | `var(--tabs-height) + var(--actions-height)` | `10` | `#f8fafc`（底色不透明） |

**关键原则**：每一层的 `top` 值 = 其上方所有 sticky 层高度之和。`z-index` 从上到下递减，确保上层始终覆盖下层。

### 13.2 核心约束：overflow 与 sticky 的冲突

```
⚠️ `overflow: hidden` 会创建新的滚动上下文，导致其内部的 `position: sticky` 元素
    相对于该容器而非外层滚动容器定位——即 sticky 失效。
```

*   **外层容器**（如 `.ledger-content` 白卡片）必须设为 `overflow: visible`，不能使用 `overflow: hidden`。
*   **如需 border-radius 裁切**：在 sticky 元素自身上添加 `border-radius`（如表头添加 `border-radius: 16px 16px 0 0`），而不是依赖父容器的 `overflow: hidden` 来做裁切。
*   `overflow: clip` 在部分场景下可替代 `hidden`（不创建滚动上下文），但浏览器兼容性需测试。

### 13.3 动态高度测量（JS 运行时注入 CSS 变量）

由于各 sticky 层的实际高度可能因内容、子环节标签等动态变化，**禁止硬编码 `top` 值**。正确做法：

```vue
<script setup>
import { onMounted, ref, nextTick } from 'vue'

const pageRef = ref(null)
const tabsRef = ref(null)
const actionsRef = ref(null)

onMounted(async () => {
  await nextTick()
  const tabsH = tabsRef.value?.$el?.offsetHeight ?? 57
  const actionsH = actionsRef.value?.offsetHeight ?? 50
  const root = pageRef.value
  root.style.setProperty('--ledger-tabs-height', `${tabsH}px`)
  root.style.setProperty('--ledger-actions-height', `${actionsH}px`)
  root.style.setProperty('--ledger-sticky-table-top', `${tabsH + actionsH}px`)
})
</script>
```

CSS 中引用这些变量：

```css
.tabs-wrapper   { position: sticky; top: 0; z-index: 20; }
.actions-bar    { position: sticky; top: var(--ledger-tabs-height, 57px); z-index: 15; }
.table-header   { position: sticky; top: var(--ledger-sticky-table-top, 107px); z-index: 10; }
```

### 13.4 Scoped CSS 穿透子组件

当 sticky 规则需要应用到子组件内部元素时，必须使用 Vue 3 的 `:deep()` 穿透选择器：

```css
/* 在父组件的 scoped CSS 中穿透到子组件的 .opportunity-list-header */
.ledger-content :deep(.opportunity-list-header) {
  position: sticky;
  top: var(--ledger-sticky-table-top, 107px);
  z-index: 10;
  border-radius: 16px 16px 0 0; /* 补偿父容器不再 overflow:hidden */
}
```

### 13.5 防止内容泄露（滚动穿透遮挡）

*   每个 sticky 层必须拥有**不透明的实色背景**，否则滚动行会透过半透明背景"泄露"出来。
*   如果 sticky 层与下方内容之间存在 `gap`（如 Flexbox `gap-4`），该间隙不会被遮挡。解决方案是让 sticky 层自身的背景向下延伸（通过 `padding-bottom`）或使用 `::after` 伪元素补偿。
*   `margin: -24px` 抵消布局 padding 的场景中，sticky `top: 0` 对应的是滚动容器的视口顶部，不需要额外偏移。

### 13.6 参考实现

台账中心 (`LedgerCenter.vue`) 为本规范的标准参考实现：
*   **页面组件**：`src/views/LedgerCenter.vue` — 动态测量 + CSS 变量注入
*   **页面样式**：`src/views/styles/LedgerCenter.css` — 三层 sticky 定义
*   **表格样式**：`src/components/ledger/styles/OpportunityList.css` — 表头基础样式（不含 sticky，由父组件注入）

## 14. 数据导入中心卡片与清单规范（以本节为准）

> 若本节与前文存在冲突，涉及数据导入中心首页、导入来源卡、导入日志表、数据总览卡等场景时，统一以本节为准。

### 14.1 数据卡片的黄金比例
*   **核心数值必须成为第一视觉中心**：数据卡片中的主数值（Hero Value）使用 `28px - 36px`，字重 `600` 或 `700`，并强制启用 `font-variant-numeric: tabular-nums;`。
*   **主数值颜色必须足够克制且有力量**：默认使用最深的中性色，如 `#1D2129`；只有真正需要预警的卡片才允许将核心数值替换为强调色。
*   **单位必须退后**：`万 / 元 / %` 等单位只能使用 `12px - 14px / 400`，颜色使用次级灰（如 `#86909C`），必须紧贴主数值右侧，并做基线对齐或轻微下沉对齐。
*   **标题只做导航，不与数值抢戏**：卡片标题统一使用 `14px / 500`，颜色使用 `#4E5969` 或 `#1D2129`，禁止用 `600+` 抢主视觉。
*   **辅助数据必须降级表达**：同环比、说明、时间戳等辅助信息统一使用 `12px - 13px / 400`，主色为灰；涨跌或异常允许局部使用柔和红绿（推荐 `#F53F3F`、`#00B42A`）做提示。

### 14.2 卡片不是死板方块
*   **卡片应具备微件感**：卡片允许使用柔和渐变底、轻边框和低对比阴影，但禁止厚重边框、死白背景和僵硬矩形块。
*   **hover 反馈统一轻浮起**：鼠标悬停卡片时，统一使用 `transform: translateY(-2px)`，并叠加更高级的扩散阴影；禁止过重的放大或夸张跳跃动画。
*   **图标只做陪衬**：当卡片中同时有图标和主数值时，图标尺寸、饱和度、占位必须退后，保证主数值始终压住画面。

### 14.3 空状态与 WIP 表达
*   **禁止突兀灰块**：空状态不得使用大面积生硬灰色底块，也禁止直接摆放刺眼的“建设中”字样。
*   **使用柔和骨架化空状态**：推荐使用 `30%` 左右透明度的浅底 + 低对比骨架纹理 / 渐变，与页面背景自然融合。
*   **图标与文案都要降噪**：空状态图标尺寸和对比度需低于正常功能卡，标题使用 `14px / 500`，说明文字使用 `12px - 13px / 400` 的柔和灰。

### 14.4 现代财务清单表格（反 Excel 感）
*   **绝对禁止竖向分割线**：表头和表体都不允许出现垂直边框，必须彻底去除 Excel 式网格切割。
*   **只保留极浅横线**：表格分隔只使用浅色水平线，推荐 `#F2F3F5`，用于引导横向阅读。
*   **最后一行不带底边框**：表体最后一行或卡片底部的清单终点，不应再有多余下边框，应直接与卡片底边融合。
*   **表头统一中性深色**：禁止彩虹表头，统一使用深灰/黑灰，字号 `13px - 14px`，字重 `400`，形成专业清单感。
*   **表体默认 Regular**：普通数据使用 `13px - 14px / 400`，禁止把整表做成半粗体。

### 14.5 表格对齐与数字格式
*   **文本列永远左对齐**：名称、说明、来源、操作人等文本型字段固定左对齐。
*   **数字列永远右对齐**：金额、计数、比例等数字字段，表头与单元格都必须 `100%` 右对齐。
*   **数字强制等宽**：所有金额、数量、日期时间码类字段，统一开启 `font-variant-numeric: tabular-nums;`，保证小数点和位数严格垂直对齐。
*   **金额强制两位小数**：涉及金额、规模、记录条数等财务型数字时，统一格式化为两位小数，如 `0.00`、`15.30`。

### 14.6 颜色语义与强调规则
*   **颜色只给状态，不给结构**：结构信息（表头、标题、常规正文）只使用中性色；红绿等状态色只能给差异、预警、异常、上涨/下降等真实语义。
*   **有偏差时才高亮**：差异/波动行若出现偏差（如 `-5.0`），可使用红色或主题强调色，并允许提升至 `600` 以形成警告感。
*   **无偏差必须降噪**：如 `0.00` 这类“无异常”值，统一使用浅灰 muted 文本，避免无意义信息抢视觉。
*   **合计行只做终极收口**：表格合计行数字使用 `600`，但避免额外花哨色彩；强调靠字重、浅底和上边界完成，而不是靠彩色染满一整行。

## 15. 导入审计详情页与通用返回控件规范（以本节为准）

> 若本节与前文存在冲突，涉及数据导入中心审计详情页、年初始化头部、台账中心返回入口、审计分页区等场景时，统一以本节为准。

### 15.1 导入审计详情页的头部与概览层级
*   **返回按钮统一复用，不允许各页各写一套**：所有页面级返回入口统一使用 `src/components/common/BackIconButton.vue`，禁止再次在页面或头部组件里手写单独样式。
*   **返回按钮视觉以台账中心为标准**：使用圆形幽灵按钮，尺寸 `34px`，浅边框、白底、轻阴影；hover 时仅允许轻微上浮、边框变蓝和浅蓝底反馈，避免做成实心主按钮。
*   **详情页标题必须比副标题更明确**：主标题使用 `18px / 600 / #1D2129`，副标题使用 `13px / 400 / #86909C`，副标题只做来源或补充说明，不允许和主标题争抢层级。
*   **导出等头部操作采用胶囊微件语言**：导出按钮使用轻边框、白底、圆角胶囊和轻微 hover 浮起，避免工具按钮过于厚重。

### 15.2 审计概览卡的正确比例
*   **右侧指标卡以“上标题、下数字”为准**：若存在顺序冲突，一律采用标题在上、Hero 数字在下的结构；数字不得在标题上方。
*   **标题与数字之间必须拉开距离**：指标卡内部标题和数字的垂直间距应明显大于普通段落间距，推荐不少于 `12px`，避免信息挤成一团。
*   **指标卡内容必须垂直居中**：卡片 body 需使用居中布局，图标、标题、数字整体在可视高度中居中，不允许出现内容贴顶或发飘。
*   **指标卡必须有独立上下留白**：卡片内部上下 padding 必须明确存在，不能只靠外层 grid gap 营造呼吸感；推荐在卡片 body 内部完成。
*   **图标是锚点，不是主角**：图标保持左侧锚点作用，但视觉权重始终低于数字；图标容器与数字之间以固定间距对齐，不允许图标尺寸压过 Hero 数字。

### 15.3 审计决策表与分页区规范
*   **分页区排布固定为“左文案、右控件”**：如 `显示第 1-5 条审计记录，共 45,210 条` 这类总数说明必须固定在左侧，分页按钮固定在右侧，不允许混排或居中漂浮。
*   **分页总数文案沿用台账中心字级**：总数说明使用 `13px / 500 / #64748B`，表现为中性、可信的底部总结，不可退成发虚的附属小字。
*   **分页控件统一采用分格按钮语言**：上一页、页码、下一页均为独立圆角分格按钮，默认白底浅边框；激活态使用品牌蓝底白字；hover 仅做轻灰底或品牌色过渡。
*   **分页按钮的尺寸和圆角应与台账中心一致**：推荐 `36px` 高度、`12px` 圆角，形成统一的 B 端分页系统观感。
*   **表内所有对比数值继续强制等宽**：导入审计详情页中比较前后值、编码、记录号、总数文案等数字型内容，统一开启 `font-variant-numeric: tabular-nums;`。

## 16. 数字报表对齐与光学校正规范（以本节为准）

> 若本节与前文存在冲突，涉及多级表头、财务数字矩阵、差异对账表、合计行数字槽、个位数/双位数混排等场景时，统一以本节为准。

### 16.1 标准对齐模型优先于肉眼硬挪
*   **禁止一开始就靠 `translateX` 硬救数字对齐**：数字报表必须先建立稳定的列级对齐模型，再考虑视觉微调。
*   **优先采用“居中列槽 + 槽内右对齐”模型**：当业务要求“表头中线”和数字所在区域保持关系时，单元格整体保持居中，内部再放固定宽度的数字槽，数字在槽内右对齐。
*   **纯财务清单场景仍以整列右对齐为优先**：若用户只要求表头/数据严格右对齐，则直接采用列级 `text-align: right`，不要额外制造居中槽。

### 16.2 数字槽（metric slot）是最佳实践
*   **数字槽必须独立于单元格本体存在**：推荐使用类似 `.metric-slot` 的内层容器承载数值，不直接让文字裸跑在 `td` 里。
*   **个位数列与金额列应使用不同槽宽**：`个数`、`金额`、`比例` 等字段的字符宽度差异明显，必须拆分为不同槽宽，不能共用一套宽度规则。
*   **槽宽优先使用固定宽度或稳定字符宽度**：可使用 `px` 或 `ch`，但必须保证正文行与合计行使用同一套基准；若合计行字级不同，优先使用固定 `px` 以避免因字号变化导致槽宽漂移。
*   **槽内数字统一右对齐**：`.metric-slot__value` 必须 `width: 100%` 且 `text-align: right;`，确保不同位数共享同一右边界。

### 16.3 等宽数字是前提，不是可选项
*   **所有财务数字、个数、日期码、记录号都必须开启 `font-variant-numeric: tabular-nums;`**。
*   **表头、正文、合计行若共享同一对齐体系，必须共享同一数字字体规则**：不能出现正文开了等宽、合计没开等宽的混用情况。

### 16.4 光学校正（Optical Adjustment）只能作为最后一步
*   **`1 / 7 / 11` 这类窄字形天然会产生“看起来没对齐”的错觉**：这是光学问题，不完全是数学对齐问题。
*   **只有在标准槽宽 + 槽内右对齐 + tabular-nums 都建立完毕后，才允许做光学校正**。
*   **光学校正必须极轻且先统一后特例**：优先在数字槽层级使用 `position: relative; left: ...` 或非常小的偏移值，推荐控制在 `1px - 4px` 内；禁止出现大幅度肉眼挪动。
*   **优先先做整套字段共享的统一偏移，再决定是否需要单列补偿**：若整组金额列都感觉偏左/偏右，应使用共享变量统一调整；只有统一调整后仍存在 `个数` 这类窄列错觉时，才允许给 `count` 槽增加更小的单独补偿。
*   **统一偏移若已能满足视觉需求，应删除多余的单列补偿**：避免“整体规则”和“局部规则”互相抵消，导致某一列看起来像没动。

### 16.5 合计行必须与正文共享同一套对齐基准
*   **合计行不得另起一套宽度逻辑**：若正文数字依赖数字槽，合计行必须使用完全同一套槽宽类名与结构。
*   **合计行字号/字重变化不能破坏对齐**：当合计数字更粗或更大时，必须优先锁定槽宽，避免由于字号变化导致 `ch` 宽度漂移。
*   **若发现“正文对齐但合计偏了”**：优先排查是否使用了不同字号引起槽宽变化，而不是先怀疑表头。
*   **合计行如果参与点击或状态表达，也必须共享正文的格式化规则**：包括 `+ / -` 前缀、零值降噪、红绿业务语义、hover 提示与点击态，不能只复用布局不复用交互与语义。

### 16.6 多级表头中的数字叶子列规则
*   **一级/二级表头负责分组，叶子列负责精确对齐**：`个数 / 金额` 这类最底层叶子表头应保持居中，数字则在叶子列对应的数字槽中完成精确对齐。
*   **表头中线与数字中线的关系应通过“列槽居中”实现，而非通过单个数字直接居中**。
*   **当用户要求“表头居中，但数字自己也要对齐”时**：默认理解为“单元格居中承载数字槽，槽内数字右对齐”，而非“数字本体在单元格中心点上”。
